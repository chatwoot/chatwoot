# Upstream Sync Strategy

## Goal

Pull `chatwoot/chatwoot` updates regularly **without ever needing to reapply
Kanban code**. Conflicts must be limited to 2 files, 1 line each.

## Why this engine survives merges

| Concern              | How we avoid the conflict                                              |
|----------------------|------------------------------------------------------------------------|
| Model name collision | `ChatwootKanban::Board` (namespaced) — Chatwoot has `Conversation`, etc., no clash possible |
| Table collision      | `chatwoot_kanban_*` prefix on every table                              |
| Migration numbers    | Engine uses `20260518…` timestamps that won't clash with upstream      |
| Controller class     | `ChatwootKanban::Api::V1::BoardsController` — namespaced               |
| Route conflict       | Engine routes nested inside `mount` block — Chatwoot's `routes.rb` keeps growing but never inside our mount |
| Frontend conflict    | Vue files live in `app/javascript/dashboard/modules/kanban/` (created by symlink); upstream never touches this folder |
| Vuex registration    | Single 1-line addition to `store/index.js`                             |
| Router registration  | Single 1-line addition to `dashboard.routes.js`                        |

## Files touched in host repo

```
Gemfile                                                         (+1 line)
config/routes.rb                                                (+1 line)
app/javascript/dashboard/store/index.js                         (+2 lines)
app/javascript/dashboard/routes/dashboard/dashboard.routes.js   (+2 lines)
app/javascript/dashboard/components/layout/Sidebar.vue          (+~7 lines, optional)
```

All other files live inside `engines/chatwoot_kanban/`.

## Recommended git workflow

### One-time setup

```sh
git remote add upstream https://github.com/chatwoot/chatwoot.git
git fetch upstream
```

### Periodic sync

```sh
git fetch upstream
git checkout develop
git merge upstream/develop
# Resolve conflicts (almost always: none, or trivial Gemfile/routes line).
git push origin develop
```

### When upstream renames Chatwoot's API base controller

The engine's `BaseController` inherits from `::Api::V1::Accounts::BaseController`.
If upstream ever renames it, you update **one line** in
`engines/chatwoot_kanban/app/controllers/chatwoot_kanban/api/v1/base_controller.rb`.

### When upstream changes the Vue layout

The engine's components are self-contained and use only `dashboard/api/ApiClient`
and `dashboard/helper/URLHelper` — public, stable utilities. If these are
renamed, update 2 import lines.

## CI safety net

Add a smoke test that fails the build if engine can no longer load:

```ruby
# spec/system/engine_loads_spec.rb
require 'rails_helper'

RSpec.describe 'Chatwoot Kanban engine' do
  it 'is mounted at the expected path' do
    expect(Rails.application.routes.named_routes.helper_names).to include(
      'kanban_boards_path'  # generated by engine routes
    )
  end

  it 'defines the expected models' do
    expect(defined?(ChatwootKanban::Board)).to be_truthy
    expect(defined?(ChatwootKanban::Card)).to be_truthy
  end
end
```

This runs in ~50ms and catches breaking upstream changes early.

## Anti-patterns (don't do these)

- ❌ Adding columns to `conversations` or `users` from the engine — those tables
  are upstream territory. Reference them via FK only.
- ❌ Reopening / monkey-patching Chatwoot models. Use the dispatcher events
  instead (`Rails.configuration.dispatcher`).
- ❌ Editing `enterprise/` for Kanban code. Use the engine.
- ❌ Inlining Kanban routes into `config/routes.rb`. Use the engine's
  `config/routes.rb` and the single mount line.
