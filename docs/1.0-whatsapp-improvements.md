## 1. Profile Picture Not Being Fetched

The WhatsApp Cloud API provides profile picture URLs via the contacts endpoint. Looking at `Whatsapp::IncomingMessageBaseService`, the contact creation doesn't fetch the profile photo. The fix is to add avatar fetching when a contact is first created:

````ruby
// ...existing code...
  def set_contact_from_message
    contact_params = processed_params[:contacts]&.first
    return if contact_params.blank?

    contact_inbox = ::ContactInboxWithContactBuilder.new(
      source_id: processed_params[:messages].first[:from],
      inbox: inbox,
      contact_attributes: { name: contact_params.dig(:profile, :name), phone_number: "+#{processed_params[:messages].first[:from]}" }
    ).perform

    @contact_inbox = contact_inbox
    @contact = contact_inbox.contact
    fetch_whatsapp_profile_picture
  end

  def fetch_whatsapp_profile_picture
    return unless @contact.avatar.blank?
    return unless inbox.channel.provider == 'whatsapp_cloud'

    Avatar::AvatarFromWhatsappJob.perform_later(@contact, inbox.channel)
  end
// ...existing code...
````

Create the new job:

````ruby
class Avatar::AvatarFromWhatsappJob < ApplicationJob
  queue_as :default

  def perform(contact, channel)
    phone_number = contact.phone_number&.delete('+')
    return if phone_number.blank?

    api_version = GlobalConfigService.load('WHATSAPP_API_VERSION', 'v22.0')
    phone_number_id = channel.provider_config['phone_number_id']
    api_key = channel.provider_config['api_key']
    base_url = ENV.fetch('WHATSAPP_CLOUD_BASE_URL', 'https://graph.facebook.com')

    response = HTTParty.get(
      "#{base_url}/#{api_version}/#{phone_number_id}/contacts",
      headers: { 'Authorization' => "Bearer #{api_key}", 'Content-Type' => 'application/json' },
      body: { blocking: 'wait', contacts: ["+#{phone_number}"] }.to_json
    )

    return unless response.success?

    profile_pic_url = response.parsed_response&.dig('contacts', 0, 'profile_pic')
    return if profile_pic_url.blank?

    Avatar::AvatarFromUrlJob.perform_now(contact, profile_pic_url)
  rescue StandardError => e
    Rails.logger.warn "[WHATSAPP AVATAR] Failed to fetch profile picture: #{e.message}"
  end
end
````

> **Note:** The WhatsApp Cloud API only returns profile pictures if the user's privacy settings allow it. Many users have it set to "My Contacts" or "Nobody", so you won't always get a photo. This is a Meta API limitation.

## 2. Improving the Flow Response Bubble UI

The current WhatsAppFlowResponse.vue is misaligned. Here's an improved version that matches the regular message bubble styling:

````vue
<script setup>
import { computed } from 'vue';
import BaseBubble from 'next/message/bubbles/Base.vue';
import { useMessageContext } from '../provider.js';
import { useI18n } from 'vue-i18n';

const { t } = useI18n();
const { content, contentAttributes } = useMessageContext();

const flowResponse = computed(
  () => contentAttributes.value?.flowResponse || {}
);

const hasFlowData = computed(
  () => Object.keys(flowResponse.value.responseData || {}).length > 0
);

const responseEntries = computed(() =>
  Object.entries(flowResponse.value.responseData || {}).filter(
    ([key]) => key.toLowerCase() !== 'flow_token'
  )
);

function formatKey(key) {
  return key.replace(/([A-Z])/g, ' $1').replace(/[_-]/g, ' ').replace(/\b\w/g, c => c.toUpperCase()).trim();
}

function formatValue(value) {
  if (value === null || value === undefined) return '—';
  if (typeof value === 'boolean') return value ? '✓' : '✗';
  return String(value).replace(/_/g, ' ');
}
</script>

<template>
  <BaseBubble class="px-4 py-3" data-bubble-name="whatsapp-flow-response">
    <!-- Message content -->
    <p v-if="content" class="text-sm text-n-slate-12 mb-2">
      {{ flowResponse.body || content }}
    </p>

    <!-- Flow response data -->
    <div v-if="hasFlowData" class="mt-1">
      <div
        class="flex items-center gap-1.5 mb-2 text-xs font-medium text-n-teal-11"
      >
        <span class="i-lucide-git-branch size-3.5" />
        <span>{{ t('CONVERSATION.FLOW_RESPONSE') }}</span>
      </div>

      <dl class="space-y-1">
        <div
          v-for="[key, value] in responseEntries"
          :key="key"
          class="flex gap-2 text-sm"
        >
          <dt class="text-n-slate-9 font-medium min-w-0 shrink-0">
            {{ formatKey(key) }}:
          </dt>
          <dd class="text-n-slate-12 min-w-0 break-words">
            {{ formatValue(value) }}
          </dd>
        </div>
      </dl>
    </div>
  </BaseBubble>
</template>
````

Key improvements:
- Uses `<dl>` definition list for semantic data display
- Inline key-value pairs instead of a bordered card grid (matches chat bubble density)
- Filters out `flow_token` (internal, not useful for agents)
- Formats camelCase/snake_case keys into human-readable labels
- Formats values (replaces underscores, handles booleans/nulls)
- Compact header with smaller icon

## 3. Where to Store Flow Response Data

**My recommendation: Store in Contact's `custom_attributes`** and also **aggregate in Conversation's `custom_attributes`**.

Here's why and the comparison:

| Option | Pros | Cons |
|--------|------|------|
| Contact attributes | Persists across conversations, searchable, visible in sidebar | Overwrites on repeat submissions |
| Contact notes | Timestamped, append-only | Not structured/searchable |
| Conversation custom_attributes | Scoped to conversation, visible in sidebar | Lost after conversation ends |
| **New "Flow Responses" tab** | Dedicated UI, full history | More code to maintain |

**Best approach:** Write to both contact `custom_attributes` (latest values) AND create a contact note (history/audit trail):

````ruby
// ...existing code...
  def store_flow_response(message)
    nfm_reply = message.dig(:interactive, :nfm_reply)
    return unless nfm_reply

    response_data = parse_nfm_response_json(nfm_reply[:response_json])
    @message.content_attributes = @message.content_attributes.merge(
      'flow_response' => {
        'body' => nfm_reply[:body],
        'name' => nfm_reply[:name],
        'response_data' => response_data
      }
    )

    persist_flow_data_to_contact(response_data) if response_data.present?
  end

  def persist_flow_data_to_contact(response_data)
    return if @contact.blank? || response_data.blank?

    clean_data = response_data.except('flow_token', 'FlowToken')

    # Update contact custom_attributes with latest flow values
    merged = (@contact.custom_attributes || {}).merge(clean_data)
    @contact.update!(custom_attributes: merged)

    # Create a note for audit trail
    note_content = clean_data.map { |k, v| "**#{k.humanize}:** #{v}" }.join("\n")
    @contact.notes.create!(
      content: "📋 WhatsApp Flow Response\n\n#{note_content}",
      user: nil,
      account: @inbox.account
    )
  end
// ...existing code...
````

This way:
- Agents see the latest data in the **Contact sidebar** (custom attributes)
- Full history is in **Contact Notes** (timestamped, searchable)
- The message bubble still shows the inline response for context

## 4. WhatsApp Integration Improvement Brainstorm

Here are actionable improvements ranked by impact:

### High Impact
1. **Read receipts sync** — The `mark_as_read` method exists but isn't automatically called when an agent reads a conversation. Auto-mark incoming messages as read when the agent opens the conversation.

2. **Typing indicators** — `send_typing_indicator` exists but isn't wired to the UI. Show typing when agent starts typing, and show "typing..." in the conversation when the contact is typing (via webhooks).

3. **Template message preview in composer** — When agents send templates, show a rich preview of what the contact will see (header image, buttons, body with variables filled in) instead of just the template name.

4. **Conversation-level WhatsApp session timer** — Show a visible countdown of the 24h session window in the conversation header. When expired, auto-switch the composer to template-only mode. This prevents failed message sends.

### Medium Impact
5. **Reaction support in UI** — `send_reaction` exists in the backend. Wire emoji reactions from the message context menu to this endpoint.

6. **WhatsApp Flow builder improvements** — The current FlowBuilder could use screen-to-screen navigation visualization, drag-and-drop reordering, and a live preview panel using the Meta preview URL.

7. **Contact phone number validation** — Before creating new WhatsApp conversations, validate the phone number format using Meta's contacts API to avoid failed message deliveries.

8. **WhatsApp catalog/product messages** — Support for product list messages and single product messages (Meta's Commerce API).

### Lower Priority but Valuable
9. **Channel health dashboard widget** — The `HealthService` data could be shown as a persistent badge/widget in the inbox settings, alerting admins when quality rating drops.

10. **Bulk template sending improvements** — The `OneoffCampaignService` could support scheduling, A/B testing of templates, and delivery rate analytics.

11. **WhatsApp Business profile management** — Allow editing the WhatsApp Business profile (about, description, address, website) from within the app settings.

12. **Message failure reasons** — When messages fail, show the specific Meta error code and human-readable explanation instead of a generic "failed" status.
