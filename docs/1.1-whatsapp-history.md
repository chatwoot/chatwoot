

  

# WhatsApp Coexistence — Implementation Guide

> **Status: IMPLEMENTED & DEPLOYED** (March 2026)

Meta's official name is **"Onboarding WhatsApp Business App users (aka Coexistence)"**. It allows a business to use **both** the WhatsApp Business App **and** the Cloud API simultaneously on the **same phone number**. Messages are mirrored between both apps.

**Doc reference:** https://developers.facebook.com/documentation/business-messaging/whatsapp/embedded-signup/onboarding-business-app-users

---

## What it does

After connecting, AirysChat:
1. **Syncs up to 6 months of message history** from the WhatsApp Business App
2. **Syncs all contacts** with their display names from the Business App
3. **Mirrors messages in real-time** — messages sent from the WhatsApp Business App appear in AirysChat and vice versa

## Architecture

### Files created/modified

| File | Type | Purpose |
|------|------|---------|
| `app/services/whatsapp/history_sync_service.rb` | **New** | Processes `history` webhooks — imports up to 180 days of messages in 3 phases |
| `app/services/whatsapp/contact_sync_service.rb` | **New** | Processes `smb_app_state_sync` webhooks — syncs contacts with full names |
| `app/jobs/whatsapp/coexistence_sync_job.rb` | **New** | Initiates sync via Meta API after embedded signup onboarding |
| `app/jobs/webhooks/whatsapp_events_job.rb` | **Modified** | Routes coexistence webhook fields to appropriate services |
| `app/services/whatsapp/embedded_signup_service.rb` | **Modified** | 3-branch channel creation: reauth by inbox, reauth by phone (coexistence), or new channel |
| `app/services/whatsapp/facebook_api_client.rb` | **Modified** | Added coexistence API methods and webhook field subscriptions |
| `app/models/concerns/message_filter_helpers.rb` | **Modified** | Added `imported_from_history?` guard to prevent bot/notification triggers on imported messages |

### Webhook routing (`WhatsappEventsJob#route_webhook_event`)

```
webhook field              → handler
─────────────────────────────────────────────────────
messages (default)         → IncomingMessageWhatsappCloudService
history                    → HistorySyncService
smb_app_state_sync         → ContactSyncService
smb_message_echoes         → IncomingMessageWhatsappCloudService (outgoing_echo: true)
account_update             → Updates coexistence status (PARTNER_REMOVED, OFFBOARDED, RECONNECTED)
```

### Data flow

```
Meta WhatsApp Server
  │
  ├── history webhook ──────────► HistorySyncService
  │   (phases 0-2, chunked)         ├── ContactInboxWithContactBuilder (find/create contact)
  │                                 ├── Conversation.create! (or find existing)
  │                                 └── Message.create!(content_attributes: { imported_from_history: true })
  │
  ├── smb_app_state_sync ──────► ContactSyncService
  │   (add/remove actions)          ├── ContactInboxWithContactBuilder (find/create contact)
  │                                 └── Contact.update!(name: display_name)
  │
  └── smb_message_echoes ──────► IncomingMessageWhatsappCloudService (outgoing_echo: true)
      (real-time mirror)            └── Message.create!(message_type: :outgoing, content_attributes: { external_echo: true })
```

## History sync details

Messages arrive in **3 phases**:
- **Phase 0**: Day 0 → Day 1 (most recent)
- **Phase 1**: Day 1 → Day 90
- **Phase 2**: Day 90 → Day 180

Each phase may have multiple **chunks** (ordered by `chunk_order`). A `progress: 100` means sync is complete. A single webhook can contain thousands of messages.

The service handles:
- Text, image, video, audio, voice, document, sticker, location, contacts message types
- `media_placeholder` messages are skipped (no downloadable content)
- Media attachments are downloaded via the Cloud API (only available for last 14 days)
- Duplicate prevention via `source_id` uniqueness check
- Original timestamps preserved from WhatsApp
- Direction detection: compares `from` field against channel phone number

### Sync progress tracking

Progress is stored in `channel.provider_config['coexistence']`:

```json
{
  "coexistence": {

    "is_business_app_onboarding": true,

```

## Imported message guard

Imported history messages are tagged with `content_attributes: { imported_from_history: true }`. The `MessageFilterHelpers` concern checks this flag to prevent:

- **Bot webhook triggers** (`webhook_sendable?`) — prevents AI bots from responding to old messages
- **Slack notifications** (`slack_hook_sendable?`) — prevents flooding Slack channels
- **Agent notifications** (`notifiable?`) — prevents notification spam

This was a critical fix — without it, the AI bot (agent bot) would attempt to respond to every imported message, causing Meta API rate limit errors.

## Embedded signup flow

The `EmbeddedSignupService` has a 3-branch `create_or_reauthorize_channel` method:

1. **Reauth by inbox** (`@inbox_id` present) — Standard reauthorization flow
2. **Reauth by phone** (`@is_business_app_onboarding`) — Finds existing channel by phone number and reauthorizes (coexistence path)
3. **New channel** (default) — Creates a new channel via `ChannelCreationService`

After channel creation/reauth, if `@is_business_app_onboarding` is true, `CoexistenceSyncJob` is enqueued to call the Meta API.

## Webhook subscriptions

The `FacebookApiClient#override_waba_callback` subscribes to these fields:

```ruby
%w[messages smb_message_echoes history smb_app_state_sync account_update]
```

## Key limitations

- **Throughput capped at 20 msg/sec** for coexistence numbers (vs. 80 default)
- **Group chats** are NOT synced
- **Disappearing messages, view once, live location** are disabled for 1:1 chats
- **Broadcast lists** become read-only
- **Voice/video calls** — not supported via API
- **Unsupported countries**: Nigeria, South Africa
- Only one sync attempt per onboarding (must offboard and re-onboard to resync)
- Media messages in history only include asset IDs for the last 14 days

---

## Pricing impact

- Messages sent from the **WhatsApp Business App** remain **free**
- Messages sent via **Cloud API** (i.e., from AirysChat) follow normal Cloud API pricing
- Customer service window (24h) only applies to Cloud API messages, not Business App messages

## Known issues / future improvements

1. **CoexistenceSyncJob explicit API call** can fail with "24h window expired" if the sync is triggered outside the window. However, Meta sends history/contacts data automatically via webhooks after onboarding, making the explicit API call non-critical.
2. **Contacts from history sync** get auto-generated names (e.g., "rough-firefly-769") when the contact is new. The contact sync (`smb_app_state_sync`) updates these names with the `full_name` from the Business App, but timing depends on when Meta sends each webhook.
3. **Spec/implementation mismatch**: The spec for `override_waba_callback` stubs only `%w[messages smb_message_echoes]`, but the actual implementation sends all 5 fields. Should be updated.

## Debugging tips

- **Check coexistence status**: Look at `channel.provider_config['coexistence']` in Rails console
- **Sidekiq logs**: Search for `[WHATSAPP COEXISTENCE]` prefix — all coexistence services use this
- **Verify imported messages**: `Message.where("content_attributes->>'imported_from_history' = 'true'").count`
- **Check webhook routing**: The `extract_webhook_field` method reads `params.dig(:entry, 0, :changes, 0, :field)` to determine which service to invoke
